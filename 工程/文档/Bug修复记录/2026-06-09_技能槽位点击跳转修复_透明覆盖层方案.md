# 技能槽位点击跳转修复 — 透明覆盖层方案

**日期**: 2026-06-09  
**文件**: `CustomSkillScreen.xml`  
**状态**: ✅ 已解决

---

## 问题描述

中间面板（技能槽位配置区）中，点击技能槽位卡片无法跳转到详细技能选择界面（`SkillSelectionPopup`）。

## 尝试过的方案

### 方案1：DoNotAcceptEvents 透明化 ❌

给 ClipRect 视口 (`SkillViewport`) 和外层 Clip 容器添加 `DoNotAcceptEvents="true"`，让事件穿透到 ButtonWidget。

**结果**：无效。可能与 ScrollablePanel 的内部事件处理机制有关，部分事件在到达 ItemTemplate 之前就被拦截。

### 方案2：多 ButtonWidget 拆分 ❌

将单个 ButtonWidget 拆分为"顶栏 ButtonWidget + 内容区 ButtonWidget"，各自绑定 `Command.Click="ExecuteClick"`。

**结果**：用户撤回。可能因为 ButtonWidget 的默认交互样式（hover/press 态）导致视觉效果异常，或事件仍被上层拦截。

### 方案3：透明覆盖层 ✅

在不改变现有布局和子元素的前提下，在 ButtonWidget 的 Children 末尾追加一个透明的 ButtonWidget 覆盖层。

## 最终实现

### 修改位置

`CustomSkillScreen.xml` 第183-185行（ItemTemplate → ButtonWidget → Children 末尾）

### 修改内容

```xml
<!-- 透明覆盖层：渲染在最上层，拦截所有点击并跳转到技能选择界面 -->
<ButtonWidget WidthSizePolicy="StretchToParent" HeightSizePolicy="StretchToParent"
              Command.Click="ExecuteClick" UpdateChildrenStates="true" />
```

### 修改前结构

```
ButtonWidget (Brush="ButtonPrimary2Brush", Command.Click="ExecuteClick")
├── TextWidget @SlotLabel (DoNotAcceptEvents)
├── TextWidget @SkillName / [未装备] (DoNotAcceptEvents)
└── Widget 内容区 (DoNotAcceptEvents)
    ├── BrushWidget 图标框
    ├── ListPanel 技能描述
    └── Widget 冷却/消耗
```

### 修改后结构

```
ButtonWidget (Brush="ButtonPrimary2Brush")
├── TextWidget @SlotLabel (DoNotAcceptEvents)        ← 被覆盖层压住
├── TextWidget @SkillName / [未装备] (DoNotAcceptEvents)  ← 被覆盖层压住
├── Widget 内容区 (DoNotAcceptEvents)                 ← 被覆盖层压住
│   ├── BrushWidget 图标框
│   ├── ListPanel 技能描述
│   └── Widget 冷却/消耗
└── ButtonWidget 透明覆盖层 (新增)                     ← 最上层渲染
    WidthSizePolicy="StretchToParent"
    HeightSizePolicy="StretchToParent"
    Command.Click="ExecuteClick"
```

## 原理分析

### GauntletUI Children 渲染顺序

GauntletUI 中，`Children` 列表越靠后的元素渲染层级越高（后渲染的覆盖先渲染的）。

### 为什么透明覆盖层有效

1. **位置**：放在 Children 末尾 → 渲染在所有子元素之上
2. **尺寸**：`StretchToParent` 宽高 → 完全覆盖整个槽位卡片（`!Skill.CardHeight`）
3. **无 Brush**：不指定画刷 → 完全透明，不影响现有视觉效果
4. **Command.Click="ExecuteClick"**：绑定与原始 ButtonWidget 相同的命令

### 为什么之前的方案失败

- `DoNotAcceptEvents` 方案：ScrollablePanel 或 ListPanel 内部可能在事件分发阶段就拦截了事件，子元素的 `DoNotAcceptEvents` 无法让事件穿透这些中间层容器
- 多 ButtonWidget 拆分方案：子 ButtonWidget 的 hover/press 态会干扰外层 ButtonWidget 的视觉效果（`UpdateChildrenStates` 冲突），且事件仍可能无法到达

### 为什么覆盖层方案成功

覆盖层不依赖事件穿透，而是**主动占据事件接收的最上层位置**。用户点击任何位置，事件首先到达覆盖层 ButtonWidget → 触发 `ExecuteClick` → 跳转到技能选择弹窗。

## 关键经验

1. **GauntletUI 中，当事件穿透不可靠时，用"覆盖层"反向解决** — 不要试图让事件穿透到目标，而是在目标位置上方放一个透明接收器
2. **Children 后面的元素渲染在上层** — 这是 GauntletUI 的布局规则，可以利用这一点实现覆盖层
3. **透明 ButtonWidget 无需 Brush** — 不指定 Brush 即为完全透明，不影响下方元素的视觉呈现
4. **`UpdateChildrenStates="true"`** 保留在覆盖层上，确保 hover/press 态能正确传递给外层 ButtonWidget（虽然覆盖层本身透明，但 UpdateChildrenStates 会将状态同步给父级，使外层卡片保持按钮交互反馈）
