# 词缀模板回退污染修复 + 日志路径统一

**日期:** 2026-06-14
**任务:** 修复同一底材不同实例（背包/商店）显示相同词缀的 bug

## 问题根因

日志分析发现 `GetAffixForEquipmentElement` 的模板回退逻辑导致污染：

```
[UI_Getter] item=peasant_pitchfork_2_t1, modifierId='bent_cheap'       → 覆盖为 锋利之愚人的干草叉
[UI_Getter] item=peasant_pitchfork_2_t1, modifierId='well_crafted_cheap' → 覆盖为 锋利之愚人的干草叉
[UI_Getter] item=peasant_pitchfork_2_t1, modifierId='null'               → 覆盖为 锋利之愚人的干草叉
```

所有 `peasant_pitchfork_2_t1` 实例（不论原版 modifier）都被 `GetAffixByBaseItemId` 模板回退查到了同一个词缀。

## 修改内容

### 1. `GetAffixForEquipmentElement` - 移除模板回退

**文件:** `EquipmentAffixSystem\Campaign\AffixCampaignBehavior.cs` (第 683-684 行)

**Before:**
```csharp
// 优先级2：模板回退
#pragma warning disable CS0618
return GetAffixByBaseItemId(element.Item.StringId);
#pragma warning restore CS0618
```

**After:**
```csharp
// 非 zzzf 词缀或无修饰符 → 不降级，避免用模板查找污染原版物品名称
return null;
```

逻辑：仅当 `ItemModifier.StringId` 能在 `ModifierToInstanceMap` 精确匹配时才返回词缀。原版 modifier（bent_cheap 等）或无 modifier 直接返回 null。

### 2. 日志路径统一到 `工程/`

4 处日志路径定义全部从动态路径改为硬编码：
```
E:\SteamLibrary\steamapps\common\Mount & Blade II Bannerlord\Modules\New_ZZZF\工程\affix_debug.log
```

| 文件 | 类 | 前缀标签 |

| `AffixUIPatches.cs` | AffixItemDescriptionGetterPatch | `[UI_Getter]` |

| `AffixDebugPatches.cs` | AffixDebug_GetModifiedItemName | `[GetModName]` |

## 预期效果

- 原版 modifier 物品（bent_cheap/well_crafted_cheap 等）保持原版名称
- 只有真正绑定了 zzzf 词缀 modifier 的物品才显示词缀名
- 同一底材的不同实例不再被同一个词缀污染

## 调试复盘 — 为何超过 3 次才定位

### 时间线

| 轮次 | 做了什么 | 为什么失败 |
|------|---------|-----------|
| 1 | 直接审查 `GetAffixForEquipmentElement` 代码逻辑 | 静态分析看不出运行时调用链全貌 |
| 2 | 排查 `ModifierToInstanceMap` 注册逻辑 + 猜可能是 Dictionary 未刷新 | 方向正确但缺证据，不敢动 |
| 3 | 检查 `AffixItemDescriptionGetterPatch` UI 调用链 | 仍在静态推断，不确定到底走了哪个分支 |

**第 4 轮才想到加日志**，日志一打立刻暴露：

```
[UI_Getter] modifierId='bent_cheap'       → 被模板回退污染
[UI_Getter] modifierId='well_crafted_cheap' → 被模板回退污染  
[UI_Getter] modifierId='null'               → 被模板回退污染
```

真相：`ModifierToInstanceMap` 精确匹配失败 → 落入 `GetAffixByBaseItemId` 模板回退 → 所有同模板物品共享词缀名。

### 教训

**超过 3 次尝试仍未定位的 bug，第 4 次强制先加日志再动代码。**
静态代码审查有盲区，运行时日志才是铁证。
