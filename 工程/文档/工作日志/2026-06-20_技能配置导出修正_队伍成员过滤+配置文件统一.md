# 技能配置导出修正：队伍成员过滤 + 配置文件统一

**日期**: 2026-06-20
**任务**: 修正技能配置导出逻辑，统一配置文件路径
**模组**: New_ZZZF

---

## 一、任务背景

之前的技能配置导出存在两个关键问题：

1. **队伍成员NPC被错误导出**：Ctrl+E 导出时，会把玩家家族中的队伍成员（包括主角、同伴、家族成员）的技能配置也写入 XML 文件，但这些 NPC 只存在于存档中，不应该作为静态配置导出
2. **配置文件冗余**：存在 `save_troop_skills.xml` 和 `ModuleData/troop_skills.xml` 两个配置文件，`save_troop_skills.xml` 只写不读，实际上从未被加载使用

## 二、解决目标

- Ctrl+E 导出的 `troop_skills.xml` 仅包含：非队伍领主NPC + 纯兵种模板
- 队伍成员NPC的技能配置仅通过存档机制保留，不出现在静态XML中
- 删除冗余文件 `save_troop_skills.xml`，统一使用 `ModuleData/troop_skills.xml`

## 四、详细修改过程

### 4.1 SkillConfigManager.cs - 新增过滤导出重载

**文件路径**: `New_ZZZF/工程/New_ZZZF/Systems/SkillConfigManager.cs`

新增带 `excludeTroopIds` 参数的 `SaveToXml` 重载：

```csharp
// 无参重载 - 全量导出（存档时使用）
public void SaveToXml(string xmlPath)
{
    SaveToXmlInternal(xmlPath, null);
}

// 带过滤重载 - 按需导出（Ctrl+E 时使用）
public void SaveToXml(string xmlPath, HashSet<string> excludeTroopIds)
{
    SaveToXmlInternal(xmlPath, excludeTroopIds);
}

// 内部实现
private void SaveToXmlInternal(string xmlPath, HashSet<string> excludeTroopIds)
{
    XDocument doc = new XDocument(new XElement("TroopSkills"));

    foreach (var kvp in _troopSkillMap)
    {
        string troopId = kvp.Key;

        // 跳过被排除的ID（队伍成员NPC）
        if (excludeTroopIds != null && excludeTroopIds.Contains(troopId))
            continue;

        // ...正常序列化XML...
    }
    doc.Save(xmlPath);
}
```

**关键点**：通过 `excludeTroopIds.Contains(troopId)` 跳过队伍成员NPC，其余条目正常写入。

### 4.2 CustomSkillScreenVM.cs - ExecuteExport 过滤逻辑

**文件路径**: `New_ZZZF/工程/New_ZZZF/GUI/CustomSkillScreenVM.cs`

在 `ExecuteExport()` 方法中新增完整的过滤逻辑：

```csharp
public void ExecuteExport()
{
    // 1. 收集排除ID = 玩家家族成员
    var excludeIds = new HashSet<string>();
    var playerClan = Clan.PlayerClan;
    if (playerClan != null)
    {
        int comeOfAge = Campaign.Current.Models?.AgeModel?.HeroComesOfAge ?? 18;
        foreach (var hero in playerClan.Heroes)
        {
            if (hero == null || !hero.IsAlive || hero.Age < comeOfAge) continue;
            if (hero.HeroState != Hero.CharacterStates.Active && hero != Hero.MainHero) continue;
            string tid = hero.CharacterObject?.StringId;
            if (!string.IsNullOrEmpty(tid)) excludeIds.Add(tid);
        }
    }

    // 2. 收集兵种白名单 = 所有纯兵种模板（非Hero、非PlayCharacter）
    var troopTemplateIds = new HashSet<string>();
    var allCOs = MBObjectManager.Instance?.GetObjectTypeList<CharacterObject>();
    if (allCOs != null)
    {
        foreach (var co in allCOs)
        {
            if (co != null && !co.IsHero && !co.IsPlayerCharacter && !string.IsNullOrEmpty(co.StringId))
                troopTemplateIds.Add(co.StringId);
        }
    }

    // 3. 关键：从排除列表中移除兵种模板（兵种模板总是导出）
    excludeIds.ExceptWith(troopTemplateIds);

    // 4. 带过滤导出
    string exportPath = "../../Modules/New_ZZZF/ModuleData/troop_skills.xml";
    SkillConfigManager.Instance.SaveToXml(exportPath, excludeIds);
}
```

**过滤逻辑三步走**：

```
步骤1: excludeIds = {主角, 同伴A, 同伴B, 领主X, ...}
         ↓
步骤2: troopTemplateIds = {commander_1, imperial_legionary, ...} (所有纯兵种)
         ↓
步骤3: excludeIds.ExceptWith(troopTemplateIds)
       → 如果某个家族成员同时也是兵种模板，则从排除列表中移除
       → 最终 excludeIds = {纯队伍NPC（非兵种）}
```

## 五、统一后的数据流

### 加载流程

```
新游戏 / L键
  → LoadFromXml("ModuleData/troop_skills.xml")
  → _troopSkillMap

游戏读档
  → HeroSkillSaveCustomBehavior.SyncData(Loading)
    → JSON反序列化到 _troopSkillMap（覆盖XML加载的数据）
```

### 导出流程

```
Ctrl+E 导出（用户手动）
  → ExecuteExport()
    → excludeIds = 队伍成员ID - 兵种模板ID
    → SaveToXml("ModuleData/troop_skills.xml", excludeIds)
    → 仅导出：非队伍领主 + 纯兵种模板

游戏存档（自动）
  → HeroSkillSaveCustomBehavior.SyncData(Saving)
    → JSON序列化到存档文件（全量）
    → SaveToXml("ModuleData/troop_skills.xml")  // 全量无过滤
```

### 对比总结

| 时机 | 触发方式 | 目标文件 | 是否过滤 | 用途 |
|------|----------|----------|----------|------|
| 新游戏 | 自动 | `ModuleData/troop_skills.xml` | 无（一次性加载全部） | 初始配置 |
| L键 | 手动 | `ModuleData/troop_skills.xml` | 无（一次性加载全部） | 热重载 |
| 游戏存档 | 自动 | 存档JSON + `troop_skills.xml` | 无（全量保存） | 持久化 |
| Ctrl+E | 手动 | `ModuleData/troop_skills.xml` | **有（过滤队伍成员）** | 导出发布 |

## 六、关键技术点

### 6.1 如何区分兵种、领主NPC、队伍成员NPC

| 类型 | `CharacterObject.IsHero` | `CharacterObject.IsPlayerCharacter` | 来源 |
|------|--------------------------|-------------------------------------|------|
| 纯兵种模板 | `false` | `false` | `MBObjectManager.GetObjectTypeList<CharacterObject>()` |
| 领主NPC | `true` | `false` | `Clan.PlayerClan.Heroes` 之外 |
| 队伍成员NPC | `true` | `true`(主角) 或 `false`(同伴) | `Clan.PlayerClan.Heroes` |

### 6.2 兵种白名单的作用

`excludeIds.ExceptWith(troopTemplateIds)` 确保即使某个家族成员的 `CharacterObject.StringId` 与兵种模板的ID冲突（虽然罕见），兵种模板仍然会被导出。

### 6.3 为什么 save_troop_skills.xml 只写不读？

历史遗留问题：创建时只在 `HeroSkillSaveCustomBehavior` 中写入，但 `SubModule` 和热重载逻辑始终加载的是 `ModuleData/troop_skills.xml`，导致 `save_troop_skills.xml` 从未被实际使用。

## 七、经验教训

1. **配置文件统一原则**：同一个数据只应该有一个配置文件来源，冗余文件只会造成混淆
2. **导出过滤的必要性**：导出是面向"发布/共享"场景，玩家私有的队伍成员数据不应该导出为静态配置
3. **兵种概念的区分**：在 Bannerlord 中，`CharacterObject` 既可以是动态的英雄（Hero），也可以是静态的兵种模板（Troop Template），过滤时需要同时考虑两者
