# 词缀模板回退污染修复 — 完整工程复盘

**日期：** 2026-06-14 凌晨  
**前置重构：** 2026-06-13 词缀系统数据模型重构（ItemRecordMap 实例层分离）  
**Bug 现象：** 同一底材的不同 modifier 实例（bent_cheap / well_crafted_cheap / 无modifier）全部显示相同的词缀名"锋利之愚人的干草叉"  
**根因：** `GetAffixForEquipmentElement` 末尾的 `GetAffixByBaseItemId` 模板回退，把原版 modifier 物品也污染了

## 一、词缀系统的完整架构

### 1.1 文件清单（13个.cs，1个系统）

```
EquipmentAffixSystem/
├── Campaign/
│   └── AffixCampaignBehavior.cs    ★ 核心协调器（45KB，本次修改的主要目标）
├── Data/
│   ├── AffixBinding.cs             装备槽绑定
│   ├── AffixDatabase.cs            词缀池（前缀/后缀定义、稀有度权重）
│   ├── AffixDefinition.cs          单条词缀定义（id, name, statModifier, weight）
│   ├── AffixedItemRecord.cs        物品实例记录包装类 ★ 6/13重构新增
│   └── AffixInstance.cs            词缀实例（前缀+后缀+属性+BuildFullName）
├── GUI/
│   └── AffixTooltipVM.cs           词缀Tooltip显示
├── Logic/
│   └── AffixGenerator.cs           词缀生成器（权重随机+冲突检查+属性结算）
├── Mission/

│   ├── AffixUIPatches.cs           ★ Harmony UI补丁（背包名称+详情面板）
```

### 1.2 三个核心字典

`AffixCampaignBehavior` 维护三个 Dictionary（都在 `SyncData()` 序列化存档）：

| 字典 | Key | Value | 用途 |
|------|-----|-------|------|
| `ItemRecordMap` | InstanceId (Guid) | AffixedItemRecord | 物品实例→词缀记录 |
| `BindingMap` | `"OwnerId:SlotIndex"` | AffixBinding | 装备槽绑定 |
| `ModifierToInstanceMap` | ItemModifier.StringId | InstanceId | **UI桥接**：ItemModifier→实例ID |

### 1.3 数据生成流程（完整链路）

```
① 游戏事件触发
   PlayerInventoryExchangeEvent / 新游戏初始化
        │
② AffixCampaignBehavior.CreateAffixedRecord(item, stackCount, source)
   ├─ ClassifyItemType(item)          分类装备类型（Weapon/Armor/Shield）
   ├─ CalculateItemLevel(item)        计算物品等级（1-100）
   ├─ new AffixedItemRecord()         创建记录，自动生成 InstanceId = Guid.NewGuid()
   │
③ AffixGenerator.GenerateSeeded(baseItemId, itemType, itemLevel, seed)
   ├─ AffixDatabase.GetAvailablePrefixes()   筛选可用前缀池
   ├─ AffixDatabase.GetAvailableSuffixes()   筛选可用后缀池
   ├─ DetermineRarity(itemLevel, roll)       判定稀有度 (Normal→Return, Magic→1, Rare→2, Unique→3)
   ├─ RollAffixes(pool, maxCount, ...)       权重随机抽取，含互斥冲突检查
   └─ ComputeFinalStats(...)                 属性叠加结算（相同stat累加）
        │
④ AffixInstance (PrefixIds + SuffixIds + FinalStatModifiers + Rarity)
   └─ record.Affix = 生成结果
        │
⑤ ItemRecordMap[record.InstanceId] = record   注册到实例表
        │
⑥ SyncAffixModifiersToPlayerRoster()
   ├─ CreateOrGetItemModifier("zzzf_affix_" + InstanceId, InstanceId)
   │   ├─ RegisterModifierBinding(modifierId, instanceId)  → ModifierToInstanceMap
   │   └─ MBObjectManager.RegisterPresumedObject(modifier)  → 注册ItemModifier
   └─ roster.Remove(old) + roster.AddToCounts(newElement, amount)
```

### 1.4 UI 显示链路（本次出问题的地方）

```
背包打开 / 物品详情
    │
    ▼
[1] Harmony补丁: AffixItemDescriptionGetterPatch (ItemVM.get_ItemDescription)
    │  拦截点：任何代码读取 ItemDescription 都会被拦截
    │  获取：element = __instance.ItemRosterElement.EquipmentElement
    │
    ▼
[2] behavior.GetAffixForEquipmentElement(element)
    │
    ├─ element.ItemModifier.StringId → "zzzf_affix_{InstanceId}"  ← 我们的词缀
    │    OR → "bent_cheap" / "well_crafted_cheap"                 ← 原版modifier
    │    OR → null                                                 ← 无modifier
    │
    ├─ 路径A: ModifierToInstanceMap.TryGetValue(modifierId, out instId)
    │   → 精确命中 → GetAffixByInstanceId(instId) → ✓ 返回正确的词缀实例
    │
    ├─ 路径B: modifierId 不在 ModifierToInstanceMap 中
    │   → 旧代码: GetAffixByBaseItemId(item.StringId)  ← 模板回退，BUG根因！
    │   → 新代码: return null                            ← 正确，不做回退
    │
    ▼
[3] affix.BuildFullName(baseItemName)
    └─ 暗黑风格命名："前缀词干之 后缀词干的 基础名"
       例："锋利之愚人的干草叉" = "锋利"之"愚人"的"干草叉"
```

### 1.5 AffixInstance.BuildFullName 命名规则

```
仅有前缀  → "残忍的 长剑"
仅有后缀  → "狼之 长剑"  
前后缀都有 → "狂战士之狼的长剑"  （前缀去"之" + "之" + 后缀去"之" + "的" + 基础名）
无词缀    → "长剑"                 （原版名称不变）
```

## 二、Bug 全过程时间线

### 第 1 轮：用户报告 "同一底材不同 modifier 显示相同词缀名"

**做了什么：**
- 审查 `GetAffixForEquipmentElement` 的代码逻辑（第660-684行）
- 看到两段：优先级1 ItemModifier桥接 + 优先级2 模板回退

**为什么没定位：**
- 静态审查看不出运行时哪段代码实际被执行
- 理论上 ItemModifier 桥接逻辑看起来是对的
- 不知道前端传入的 modifierId 是什么值

### 第 2 轮：排查 ModifierToInstanceMap 注册逻辑

**做了什么：**
- 检查 `CreateOrGetItemModifier` 和 `RegisterModifierBinding`
- 检查 `SyncAffixModifiersToPlayerRoster` 同步逻辑
- 怀疑是 Dictionary 未刷新或注册时机问题

**为什么没定位：**
- 注册逻辑没问题，方向正确但缺证据
- 不知道实际运行时 Map 里有什么、缺什么

### 第 3 轮：检查 Harmony UI 补丁代码

**做了什么：**
- 审查 `AffixItemDescriptionGetterPatch.Postfix`（第29-72行）
- 分析 `AffixItemPreviewPatch.Postfix`
- 看调用链：ItemVM.get_ItemDescription → GetAffixForEquipmentElement → ???

**为什么没定位：**
- 仍在做静态推断，不知道实际传入的 modifierId 究竟是 `zzzf_affix_xxx` 还是 `bent_cheap`
- 不知道 ModifierToInstanceMap.TryGetValue 匹配了还是失败了

### 第 4 轮：加日志（转折点）

**做了什么（用户明确提出加日志后）：**
- 在 `AffixItemDescriptionGetterPatch.Postfix` 开头加了调试日志
- 日志输出：item.StringId, modifierId, ModifierToInstanceMap.Count, ItemRecordMap.Count, 查找结果

**日志分析结果（一击定位）：**
```
[UI_Getter] item=peasant_pitchfork_2_t1, modifierId='bent_cheap'       → TemplateFallback → 锋利之愚人的干草叉
[UI_Getter] item=peasant_pitchfork_2_t1, modifierId='well_crafted_cheap' → TemplateFallback → 锋利之愚人的干草叉
[UI_Getter] item=peasant_pitchfork_2_t1, modifierId='null'               → TemplateFallback → 锋利之愚人的干草叉
```

**真相大白：**
1. `bent_cheap` 和 `well_crafted_cheap` 是原版 modifier，不在 `ModifierToInstanceMap` 中
2. ItemModifier 桥接匹配失败 → 落入 `GetAffixByBaseItemId("peasant_pitchfork_2_t1")`
3. `GetAffixByBaseItemId` 遍历 `ItemRecordMap`，找到第一个 BaseItemId 匹配的实例
4. 该实例恰好有词缀"锋利之愚人" → 返回 → 污染了所有同底材的物品名称

## 三、修复方案

### 3.1 GetAffixForEquipmentElement — 移除模板回退

**文件：** `EquipmentAffixSystem\Campaign\AffixCampaignBehavior.cs`  
**位置：** 第 682-684 行

```diff
-            // 优先级2：模板回退
- #pragma warning disable CS0618
-            return GetAffixByBaseItemId(element.Item.StringId);
- #pragma warning restore CS0618
+            // 非 zzzf 词缀或无修饰符 → 不降级，避免用模板查找污染原版物品名称
+            return null;
```

**原理：** 只接受通过 `ModifierToInstanceMap` 精确匹配的结果。不在 Map 中的（原版 modifier / 无 modifier）直接返回 null → UI 层看到 null → 不替换名称 → 保持原版显示。

### 3.2 日志路径统一到 `工程/`

4 处日志路径全部从 `Assembly.GetExecutingAssembly().Location` 动态路径改为硬编码：

```
E:\SteamLibrary\steamapps\common\Mount & Blade II Bannerlord\Modules\New_ZZZF\工程\affix_debug.log
```

| 文件 | 类 | 日志前缀 | 旧路径方式 |
|------|-----|---------|-----------|
| AffixCampaignBehavior.cs | Campaign | `[HH:mm:ss.fff]` | Assembly.Location |
| AffixUIPatches.cs | AffixItemDescriptionGetterPatch | `[UI_Getter]` | typeof().Assembly.Location |
| AffixUIPatches.cs | AffixItemPreviewPatch | `[UI_Preview]` | typeof().Assembly.Location |
| AffixDebugPatches.cs | AffixDebug_GetModifiedItemName | `[GetModName]` | typeof().Assembly.Location |

**原因：** 旧动态路径指向 `bin\Win64_Shipping_Client\`，日志散落各处。统一到 `工程/` 方便查看。

### 3.3 XML 注释更新

`GetAffixForEquipmentElement` 的方法注释从"实例优先，模板回退"改为"仅精确实例查找"。

## 四、调试经验教训

### 核心教训：3次静态分析未定位 → 第4次强制加日志

| 轮次 | 方法 | 结果 | 问题 |
|------|------|------|------|
| 1 | 读代码审查调用链 | ❌ 不行 | 看不出运行时分支 |
| 2 | 排查注册逻辑 | ❌ 不行 | 方向对但缺证据 |
| 3 | 审查 Harmony 补丁 | ❌ 不行 | 仍在静态推断 |
| 4 | **加日志** | ✅ 一击定位 | 3 行日志揭露真相 |

**规则已入库（Memory ID: 11299692）：** 同一问题超过 3 次尝试仍未定位时，第 4 次禁止继续静态审查，必须先加日志。

### 日志应该打什么（模板）

对于此类运行时行为异常问题，日志最少应覆盖：

```
1. 入口：方法被调用时的关键参数
   └─ item.StringId, modifierId

2. 决策点：每个分支判断的结果
   └─ ModifierToInstanceMap.TryGetValue 成功/失败
   └─ GetAffixByBaseItemId 是否被调用

3. 结果：最终返回值的关键属性
   └─ affix == null / affix.Rarity / affix.BuildFullName()
```
